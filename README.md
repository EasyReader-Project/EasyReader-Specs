# 📜 EasyReader-Specs · 公共契约仓库

> **数据同源 · 双端一致 · 契约先行**

本仓库是 EasyReader Project 的**唯一事实来源**。它不包含任何端侧代码，而是存放所有跨端共享的**数据格式定义、数据库结构、常量枚举和测试用例**。

任何涉及阅读进度、书签笔记、用户配置或本地存储格式的变更，**必须先在本仓库发起 PR 并更新契约**，随后 Windows 和 Android 端才能跟进实现。

---

## 🎯 为什么需要这个仓库？

EasyReader 拥有 Windows 和 Android 两个独立的原生端。如果没有统一的“契约”，两端很容易出现以下问题：

- **进度无法互通**：A 端记录的章节索引在 B 端被解析为乱码。
- **配置键名打架**：手势动作在 Windows 叫 `TAP`，在 Android 叫 `CLICK`，导致配置备份失效。
- **数据库字段漂移**：两端各自给 `books` 表增加字段，最终无法合并。

Specs 仓库确保所有端在同一个数据协议下协作。

## 📁 目录结构说明

### 📂 `sql/` · 数据库表结构
存放 SQLite 建表语句及版本迁移脚本。

- **核心内容**：`schema.sql` 包含 `books`（书籍元数据）、`progress`（阅读进度）、`bookmarks`（书签）、`notes`（笔记）等表的完整定义。
- **约束作用**：双端必须依据此文件创建数据库，字段类型、长度、默认值保持绝对一致。
- **迁移管理**：`migrations/` 目录存放版本升级脚本，例如 `v1.0.0_to_v1.1.0.sql`，用于在不丢失用户数据的前提下升级结构。

### 📂 `schema/` · JSON 格式规范
这是**整个契约仓库的心脏**。所有以 JSON 形式存入数据库或通过 WebDAV 传输的数据，都必须符合这里的定义。

- **`progress.schema.json`**：定义三级定位（精确定位 / 语义降级 / 百分比）的 JSON 结构，是进度同步的核心。
- **`bookmark.schema.json`**：规范书签的存储字段（所属章节、高亮文本、颜色、创建时间等）。
- **`note.schema.json`**：规范笔记/批注的结构（关联的书签 ID、纯文本内容、位置偏移）。
- **`config.schema.json`**：约束用户自定义配置的存储格式，包括手势映射表、主题 CSS 路径、字体偏好等。

### 📂 `protocol/` · 通信与同步规则
用纯文本文件说明跨设备交互的行为准则，而非定义代码接口。

- **`webdav_backup_format.md`**：规定备份文件在 WebDAV 服务上的存储路径结构（如 `/EasyReader/backup_2026-07-06.json`）以及是全量覆盖还是增量合并。
- **`conflict_resolution.md`**：详细阐述 **Last-Write-Wins（最后写入胜出）** 规则，明确双端如何基于毫秒级时间戳仲裁冲突。

### 📂 `constant/` · 共用常量与枚举
存放两端需要硬编码对照的“字典”。

- **`gesture_actions.json`**：统一手势动作枚举。例如规定 `SINGLE_TAP` 代表单击，`DOUBLE_TAP` 代表双击，两端在日志和配置文件中必须使用同一套词汇。
- **`error_codes.json`**：统一错误码（如 `1001` 代表文件解析失败），方便双端上报日志时解读一致。

### 📂 `test/` · 共享测试数据集
存放用于验证双端实现是否合规的固定测试样本。

- **`fixtures/`**：放置用于测试的一本标准 EPUB 和 PDF 样本文件。
- **`progress_test_cases.json`**：涵盖各种边界场景的测试数据（如 CFI 为空时的降级表现、超大页码的容错处理），两端开发完成后必须能通过同一套测试。

## 🚀 如何使用本仓库？

1.  **开发新功能前**：先查看本仓库是否有对应的契约定义。如果没有，请先新建或修改 `schemas/` 或 `sql/` 下的文件。
2.  **提交 PR**：在修改契约的 PR 描述中，需说明变更原因以及是否破坏向后兼容性。
3.  **通知双端**：契约合并后，在 Windows 和 Android 仓库中发起对应的实现 PR，并关联本仓库的 Commit ID。
4.  **版本升级**：若变更不兼容（如删除字段或修改类型），**必须**更新 `CHANGELOG.md` 并递增主版本号。

## 🤝 贡献者须知

- 所有约定文件的格式必须清晰易读（推荐使用 JSON Schema 标准或 Markdown 表格）。
- 禁止在本仓库存放任何特定端（Win/Android）的实现代码、UI 资源或第三方依赖。
- 保持原子化提交：一次 PR 只解决一个契约问题。
